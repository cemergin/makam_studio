## TL;DR — what v1 is

- **A free, browser-based, microtonally accurate maqam instrument** with three subsystems sharing one tuning-data layer: lead synthesis, MIDI translator, drone.
- **Default playing surface:** split-keyboard, 30 % left zone (preset selector + karar slider) / 70 % right zone (microtonal piano keyboard). Mobile portrait collapses to vertical 50/50.
- **5 synthesized lead voices** — qanun, baglama, tar, ney, clarinet — sharing one Karplus-Strong / waveguide AudioWorklet kernel. Monophonic in v1.
- **24 maqam presets** — 8 each from Turkish, Arabic (Levantine + Egyptian), and Persian traditions, each cited to a primary source.
- **Drone-on-by-default**, single voice (vaporwave-choir-tubular-wind), with **Tab-to-current-pitch karar modulation** as the project's signature improvisational gesture.
- **MIDI in / MIDI out:** vanilla 12-EDO MIDI in becomes correctly-tuned MIDI out via single-channel + 14-bit pitch-bend (default ±2 semitones), plus `.scl + .kbm` file export for DAW interop.
- **Pitch-network with button-overrides:** long-press / right-click any right-zone key to open a 4–6-button radial palette of regional micropitch alternates (Cairene / Aleppine / Iraqi / Turkish Sikah, koron variants, comma adjustments).
- **PWA, offline-first, MIT-licensed, no backend, no accounts, no telemetry.** All state in IndexedDB + localStorage. Share by URL fragment.
- **Engine forked from BeatForge** per the inventory in `research/beatforge-reusable-research-mining.md`.
- **Out of scope for v1:** polyphony, bowed strings, MPE, MTS-ESP runtime, modern-synth patch category, practice mode, patch editor, Hindustani / Byzantine / Maghrebi / Azeri / Shashmaqam preset tabs, social features, audio sample import.

## 1. Identity & runtime constraints

- **Browser-based.** Deployed as a static site at `https://cemergin.github.io/makam_studio/` (or equivalent custom domain), served over HTTPS.
- **PWA.** Installable, offline-capable after first load. Workbox-driven service worker (precaches HTML/JS/CSS, fonts, all preset JSON, voice DSP code).
- **MIT license.** Source-available, contributable, no economic gating.
- **No backend, no accounts, no auth.** All persistence is local — IndexedDB (Dexie) + localStorage. Sharing happens via URL fragment, not server.
- **No telemetry, no cloud, no ads, no monetization.** What the user plays stays on their device.
- **Synthesized audio only.** No bundled samples, no sample import in v1. (Sample-free is a permanent constraint, not a phasing one — see MANIFESTO §4.)
- **Pure Web Audio API.** No Tone.js, no third-party DSP runtime. AudioWorklet for the per-sample work; vanilla AudioNodes for the master bus + FX returns.
- **React 19 + Vite 8 + TypeScript (strict).** Bun as package manager + CI runner. ESLint + Prettier. Vitest for unit tests; Playwright for smoke tests.
- **Web MIDI API.** Required for the killer feature (MIDI in → maqam-tuned MIDI out). Gracefully unavailable in Safari (see §10, §17).
- **No server-side anything.** GitHub Pages hosts static files; no edge functions, no auth callbacks, no analytics endpoints.
- **CPU target.** Mid-range phone (iPhone 12 / Pixel 6 class) must hold the full v1 patch (1 lead voice + 1 drone = ~5 K-S-equivalent voices total) without underrun. Desktop must comfortably hold v2's polyphonic ambition (~24 voices).

## 2. Layout — desktop and mobile

makam_studio ships **three layout modes**. The user toggles between them via a layout-mode icon group in the left zone (desktop) or in the gear menu (mobile). The active mode persists in `localStorage.ms_layout`.

### 2.1 The three layout modes

| Mode | Default? | Purpose | Trigger |
|---|---|---|---|
| **split-keyboard** | yes (≥1024 px desktop, portrait mobile) | Pivot 2 primary mode. Left zone selects what's available; right zone plays melody. | default; URL `?layout=split` |
| **single-surface** | no | Maximises right-zone keyboard width; preset selector hides into a bottom sheet (mobile) or popover (desktop). | layout toggle; URL `?layout=single`; landscape mobile (forced) |
| **qanun-honor** | no | Power-user view. Replaces the right-zone piano with a literal trapezoid + 26-course mandal grid. | layout toggle; URL `?layout=qanun` |

The split-keyboard layout is what the user sees on first visit. The other two are opt-in.

### 2.4 Mobile landscape (≤900 px tall, wider than tall)

Auto-collapses to **single-surface** mode. The maqam grid is hidden; a floating "M" button (top-left) opens a half-screen bottom sheet for preset selection. Keyboard expands to ~95 % of screen height. This is the "I want to play" mode — phone held landscape, two thumbs on the keyboard.

### 2.5 Hit-target minimums

All interactive surfaces meet **44 × 44 pt minimum** (Apple HIG; WCAG 2.5.5). Specifics:

- Maqam grid pads: **88 × 88 pt** (desktop), **72 × 72 pt** (mobile). Comfortably exceeds minimum.
- Karar slider thumb: 32 pt diameter; the strip itself has a 56 pt-tall hit area.
- Right-zone white-equivalent keys: 48 pt wide × 240 pt tall (desktop) / 44 pt wide × 160 pt tall (mobile).
- Right-zone microtonal accidental keys: 32 pt wide × 144 pt tall, on top of the white-equivalents (standard piano stacking).
- Octave nav, drone pill, layout toggle: 44 × 44 pt minimum.
- Per-pitch override radial menu items: 56 pt diameter at full-extension (≥80 pt from centre).

## 3. Left zone — preset selector

The left zone is the "context" half of Pivot 2's split-keyboard. It is a **silent** zone in the autoharp sense — pressing here does not produce a note; it changes what subsequent right-zone presses produce.

### 3.1 Tradition tabs (in header, but logically left-zone)

Three tabs in v1: `TR` (Turkish), `AR` (Arabic — Levantine + Egyptian merged into one tab), `FA` (Persian). Native-script subtitle on hover/long-press: `Türk` / `عربي` / `فارسی`. The active tab carries the tradition's accent colour (§18). Switching tabs replaces the maqam grid's 16 buttons (8 v1 presets + 8 "More…" slots opening a deeper catalog drawer) and resets the active maqam to the tradition's first preset (Rast / Rast / Shur).

### 3.2 Maqam preset grid

**4 × 4 grid of 16 buttons.** v1 fills 8 slots per tradition (the canonical-and-cited maqamat from §7); the remaining 8 slots are placeholders showing `+ More…` — tapping any opens a search / catalog drawer that lists the rest of the tradition's repertoire (read-only in v1, with "Coming in v2" footer for ungenerated entries).

Each pad shows:

- **Maqam name** in canonical Latin (centre, large, Inter 600).
- **Native-script subtitle** (top, smaller, tradition-correct font: Vazirmatn for Persian, IBM Plex Sans Arabic for Arabic, Inter for Turkish).
- **Region tag** for the Arabic tab (`Cai` / `Alp` / `Bgh` for Cairene / Aleppine / Baghdadi when relevant; absent on Turkish / Persian where one variant ships).
- **Active state.** Filled with the tradition's accent colour, white text. Inactive: warm-cream background, ink text, 1 px ink border.

**Tap = activate** (instantaneous retune of right zone — <10 ms latency target — with a brief 200 ms highlight pulse and an optional subtle "mandal click" sound, default off, configurable in settings).

**Long-press a pad** (≥ 500 ms) = enter Customize mode; pads become drag-reorderable; user-assigned slots persist in `ms_custom_grid_<tradition>`. Standard tap remains the activation gesture.

### 3.3 Karar slider

Below the maqam grid, full-width, 56 pt tall. Two-tier control:

- **Coarse:** chromatic 12-EDO strip showing 12 named pitches `A B♭ B C C♯ D E♭ E F F♯ G A♭`. Active karar is highlighted with the tradition accent colour. Tap to set; drag to scrub. Range A2–A4 (24 semitones), default **C4** for first-launch (subverts "C is home" minimally; users can re-default in settings).
- **Fine cents nudge:** a `± cent` micro-control immediately right of the chromatic strip. Range ±100 cents, 1-cent step, displayed numerically. Lets users slide karar off the 12-EDO grid for non-A=440 references or for matching a tape recording's pitch reference. Persists per-session.

The active karar is shown to the right of the slider in tradition-formatted prose: `Hicaz on D` / `Hijaz fî Re` / `Hijaz · D · +12 ¢`, depending on label-mode and active tab.

### 3.4 Per-pitch overrides drawer

A collapsible `▾` drawer below the karar slider. Closed by default. When open, exposes every pitch in the currently-active maqam preset with:

- The pitch's perde name (and Latin / cents alternates per active label-mode).
- Its current cents-from-karar value.
- A button row of **available regional alternates** (the candidate pitches from the per-degree palette — see §11): e.g. for Maqam Rast's third degree, `Cai 350 ¢ · Alp 340 ¢ · Iraqi 360 ¢ · Turk 384 ¢`.
- A `↺ Reset to preset default` per pitch.

This drawer is the **non-radial** path to the same data the long-press radial menu in the right zone exposes (§6, §11). Power users prefer the drawer; touch users prefer the radial. Both write to the same in-memory state.

### 3.5 Layout toggle (icon group)

Bottom of the left zone. Three 32 × 32 pt icons:

- **`⊞`** — split-keyboard (default).
- **`▭`** — single-surface.
- **`△`** — qanun-honor.

The active mode's icon is filled with coral (§18). Mouseover shows the mode name as a tooltip.

### 3.6 What the left zone deliberately is *not*

- Not a chord pad. No major / minor / 7 / 9 vocabulary.
- Not a strum surface. The grid does not play notes.
- Not an auto-accompaniment trigger. No drums; no auto-bass; no progression engine. (BeatForge owns rhythm.)
- Not a search-first interface. The 16-pad grid is the home; a `+ More…` catalog drawer is one tap away but not the default surface.

## 4. Right zone — playing keyboard

The right zone is the "articulation" half. It is a **piano-style microtonal keyboard** — Pivot 1's canonical surface, with the strings unhidden and the 12-EDO lock removed.

### 4.1 Layout, range, octave navigation

- **Visible range:** 2.5 octaves on desktop (≈18 white-equivalent keys + microtonal accidentals); **1.5 octaves on mobile portrait**; **2 octaves on mobile landscape (single-surface mode)**.
- **Default centre:** karar's octave + half above. With karar = C4, the visible window spans roughly C3 → A5.
- **Octave navigation:**
  - Two `[Octave −]` `[Octave +]` buttons top-right of the right zone (44 × 44 pt each).
  - Mouse-wheel scroll inside the zone (desktop).
  - Two-finger swipe horizontally (mobile).
  - Keyboard arrows `↑` `↓` shift by octave; `←` `→` shift by single visible key.
  - Octave shifts persist per-session in `ms_octave_offset`.

### 4.2 Key topology

The right zone preserves piano-style **white / black key visual rhythm** as the default — Pivot 1's "the piano is a kanun in a tux" framing. Each white-equivalent key carries a scale degree of the active maqam, retuned to the maqam's actual cents-from-karar. Microtonal accidentals (the keys traditionally called "black") appear in piano arrangement, with the active maqam's microtonal alternates and Pythagorean / quarter-tone neighbours rendered there.

The right zone is **not** an isomorphic grid. It is a piano with shifted pitches.

**Asymmetric maqam handling (Hüseyni, Saba, etc.):** when the active maqam declares a `pitches_descending` (per the schema in §13), the engine tracks melodic direction (each note's pitch relative to the previous one) and consults the appropriate ascending or descending pitch table. This is per-voice, not per-instrument; in v1 (monophonic), it is per-note simply.

### 4.3 Microtonal accidental rendering

The right zone renders tradition-specific microtonal symbols on the keys themselves (small, on the upper face of each key, in coral on the warm-cream key fill, or in cream on coral when the key is depressed):

| Tradition | Glyphs to render | Source |
|---|---|---|
| Turkish | AEU comma marks: `1`, `4`, `5`, `8` koma flats and sharps (rendered as small numeric superscripts adjacent to the standard `♭` `♯` glyph). | research/ottoman-turkish-makam.md §2.2 |
| Arabic | Mishaqa half-flat (`d` slashed) and half-sharp (`d` with crossbar). | research/arabic-levantine-maqam.md §1 |
| Persian | Koron (`p` reflected) and sori (`p` with stem). | research/persian-dastgah.md §2 |

**Glyph rendering** is from a small custom SVG sprite (≤ 2 KB), not from a font — fonts that include AEU comma accidentals are scarce and the schema's `display_accidental` enum (§13) keys directly into the sprite.

### 4.4 Pitch labelling modes

Toggle in the header label-mode dropdown (`[labels: perde ▾]`):

| Mode | Primary label | Secondary | When used |
|---|---|---|---|
| **perde** (default) | Perde name (`Rast`, `Dügâh`, `Çârgâh`) | tradition-localised script if the active tab is AR or FA | Default for advanced users; idiomatic. |
| **Latin** | Western letter name (`G`, `A`, `D♭`) | perde name on hover | Beginner-friendly; pianist-compatible. |
| **cents** | `+0 ¢`, `+204 ¢`, `−15 ¢ from E` | perde + Latin on hover | Tuning-engineer mode. |
| **hidden** | (no labels) | hover shows perde + Latin + cents | Performance / clean-screen mode. |

Persists in `ms_label_mode`.

### 4.5 Velocity and articulation

- **MIDI input:** velocity passes through verbatim.
- **Touch:** Pointer Events `pressure` where available (Apple Pencil, supported Android styluses); fallback fixed velocity 100/127.
- **Mouse / trackpad:** fixed velocity 100/127; click duration drives release time.
- **QWERTY:** fixed velocity 100/127; configurable in settings.
- **Sustain (CC 64):** spacebar (toggle) or sustain-pedal MIDI input. Off by default in v1.

### 4.6 Long-press = override-palette

Long-press (≥ 500 ms) on any right-zone key opens an inline **radial palette overlay** centred on the key, presenting the 4–6 candidate microtonal alternates for that scale degree. Detail in §11.

This is the bridge between piano-canonical default and qanun-honor power-user mode. It also satisfies the multiparametric-mapping criterion: a single key has two output behaviours selected by gesture duration (tap = play; long-press = retune).

### 4.7 What the right zone deliberately is *not*

- Not a strum surface (no multi-string articulation gesture).
- Not a continuous-pitch fingerboard (default; ney / oud-style continuous-pitch UX is a v2 voice-mode toggle).
- Not a sequencer / piano-roll editor.

## 6. The right-zone keyboard interaction model — explicit

(Consolidates §4 + §11 cross-references for clarity.)

| Gesture | Behaviour |
|---|---|
| Tap a key | Plays the canonical maqam pitch for that scale degree; voice = active voice; karar applies. Velocity from input layer. |
| Hold a key | Sustains for as long as held (or until sustain pedal releases). |
| Long-press (≥500 ms) | Opens override-palette radial menu over the key. The note continues sounding at the canonical pitch until a palette item is tapped or the menu is dismissed (tap-elsewhere). |
| Tap a palette item | Re-pitches that one note to the alternate; the canonical note (already sounding) glides to the new pitch over 50 ms. After the held note releases, subsequent taps on the same key revert to canonical (one-shot mode). |
| Toggle "Lock" in palette | Makes the override sticky for that scale degree until explicitly cleared or maqam changes. Lock state persists in IndexedDB with the user's session. |
| Right-click (desktop) / two-finger-tap (mobile) | Identical to long-press. |
| Double-tap a key | Reserved (unused in v1; v2 candidate for "open per-pitch editor"). |

### 6.1 Octave navigation — explicit

| Gesture | Behaviour |
|---|---|
| `[Octave −]` / `[Octave +]` button | Shifts visible window ±1 octave. Persists in `ms_octave_offset`. |
| Mouse wheel inside right zone | Continuous shift; snaps to octave boundaries on release. |
| Two-finger swipe horizontally (mobile) | Smooth shift. |
| Pinch (mobile) | Zooms 1.5 ↔ 2 octaves visible (mobile only). |

## 7. The 24 v1 maqam presets

Eight presets per tradition. Each preset names: canonical Latin · native script · transliteration · default karar (all C4 except where noted, per design-direction) · cents-from-karar for each pitch (ascending; descending given when it differs) · perde / jins composition / gusheh notes as appropriate · one-sentence character · primary reference. **Cent values use the tradition's grid:** Turkish on 53-EDO Holdrian; Arabic on 24-EDO; Persian as continuous cents (no snap), with the values chosen from Farhat / Talai measurements.

All presets have `karar = C4 ⇒ cents_offset_from_a440 = −900.0`. The actual played pitch when karar is moved follows the schema rule `Hz = 440 × 2^((karar_offset + pitch_cents_from_karar) / 1200)` (§13).

## 8. The 5 v1 lead voices

All five voices share **one AudioWorklet processor file** (multiple voice classes inside) and a unified `pluck.ts`/`bore.ts` kernel pair. Per `research/multi-instrument-synthesis.md` §1–§3.

**Default polyphony cap = 1 (monophonic v1).** New note-on cuts active note. Polyphony is v2.

### 8.1 Voice roster

| # | Voice | Synthesis | Default polyphony | CPU per voice | User-exposed parameters |
|---|---|---|---|---|---|
| 1 | **Qanun** (default) | Karplus-Strong + Thiran fractional delay; 3 detuned strings/course; 4-biquad body filter (peaks 280 / 1200 / 3500 Hz + 8 kHz HF cut). | 1 (mono) | 1.0 × baseline (~30 mults/sample) | brightness · decay · body resonance · pluck position · mandal-click level |
| 2 | **Baglama** | K-S param swap on qanun kernel; 2 unison strings (±2¢); body filter Q=4–6, fundamental ~300 Hz; β=0.15 (struck further from bridge). | 1 (mono) | 1.0 × | brightness · decay · body resonance · plectrum hardness · finger-stretch (vibrato depth) |
| 3 | **Tar** | K-S + dual-cavity body filter (peaks 190 + 340 Hz, 5 biquads total); membrane-buzz layer (–30 dB HP-noise modulated by amplitude). | 1 (mono) | 1.1 × | brightness · decay · body resonance · membrane buzz · pluck-position |
| 4 | **Ney** | Cylindrical-bore waveguide (single bidirectional delay line) + jet-edge nonlinearity + breath-noise envelope; 1-pole HP body radiation. | 1 (mono) | 2.5 × | breathiness · brightness · vibrato rate · vibrato depth · half-hole bend |
| 5 | **Clarinet** | Single-reed waveguide (Smith canonical) + reed nonlinearity + cylindrical-bore feedback + bell-radiation HP. | 1 (mono) | 3.0 × | brightness · breathiness · reed hardness · vibrato rate · embouchure-bend |

**Total CPU budget at v1 monophonic:** 1 lead voice (1.0–3.0 ×) + 1 drone voice (≈ 3 ×) ≈ 4–6 × baseline. Comfortably inside the iPhone-12 / Pixel-6 budget Wave 1 established (~6 baseline voices on phone, ~24 on desktop).

### 8.2 Per-voice pitch-mode and right-zone behaviour

| Voice | `pitchMode` | Right-zone behaviour |
|---|---|---|
| Qanun | `discrete` | Tap = canonical pitch; long-press = override palette. No pitch motion mid-key. |
| Baglama | `discrete-with-bend` | Tap = canonical pitch; small finger-stretch bend gesture (drag up/down on key) gives ±20 cents; long-press = override palette. |
| Tar | `discrete-with-bend` | Tap = canonical pitch; finger-drag = ±100 cent slide (fretless under fingering); long-press = override palette. The override palette gets heaviest use here. |
| Ney | `glide` | Tap = canonical pitch; drag-along-key = continuous glide (half-holing analogue); aftertouch / vertical drag = vibrato depth. |
| Clarinet | `glide` | Tap = canonical pitch; drag = continuous embouchure-bend; aftertouch = breath-amount (timbre). |

Voice change is independent of maqam change. Any maqam plays on any voice.

### 8.3 Master FX chain

Forked from BeatForge's `app/src/audio/machines/fx/`:

```
voice → voice gain → master bus → compressor → destination
                          ↓
                       reverb send (synthesized noise-burst impulse) → reverb return → master bus
                          ↓
                       delay send (feedback delay) → delay return → master bus
```

Drone has its own send to the reverb bus at higher wet level (60 % vs lead's 30 %).

### 9.2 Voice / interval / volume / glide-time UX

The drone strip exposes (in order, left-to-right):

1. **On/off pill** (filled when on; tradition-accent-coloured).
2. **Voice label** (read-only in v1; reads `vaporwave-choir-tubular-wind`).
3. **Interval radio** (`sa-pa` / `sa-Ma` / `single-tone` / `sa-pa-octave` / `custom`). `custom` opens a two-pitch picker.
4. **Volume slider** (0–100 %, default 30 %).
5. **Tab button** `[↻ Karar]` — primary trigger on mobile; redundant with Tab key on desktop.
6. **Modulation-mode radio** (`Re-center` / `Float`).
7. **Glide-time slider** (200–1500 ms; default 600).
8. **Custom-pitch input** — opens a popover with Hz / cents-from-A4 entry (locked off by default).
9. **`▾` Advanced** — opens the advanced patch panel (layer balance, vowel-cycle speed, formant brightness, drift on/off, tremolo rate / depth, reverb send). Power-user territory; persists in `ms_drone_patch_advanced`.

### 9.3 Tab-to-current-pitch — explicit interaction

Detailed in §12.

### 9.4 Re-center vs. Float — explicit

- **Re-center (default).** Active maqam re-anchors to the new karar. The pitch network shifts; the override-palette refreshes (Aleppine vs. Iraqi vs. Cairene Sikah candidates rebase). The MIDI translator (§10) emits a `karar_change` event.
- **Float.** Drone moves; maqam stays anchored. Keyboard mapping, override-palette, and MIDI mapping unchanged. The harmonic relationship between drone and maqam-anchor becomes the expressive variable.

A radio toggle in the drone strip; default Re-center. (v2 candidate: Shift+Tab as one-shot Float.)

### 9.5 Custom drone pitch lock

When engaged, drone holds at a user-set Hz or cents-from-A4 regardless of karar changes. Tab presses become no-ops while custom-pitch is engaged (with a toast: "Custom drone pitch locked — disable to use Tab"). Useful for tape-collaboration, fixed-pitch-reference work, or generative-drone aesthetics.

## 10. MIDI in / MIDI out flow

Per `research/midi-maqam-translation.md`. The MIDI translator is the killer feature: vanilla 12-EDO MIDI in becomes correctly-tuned MIDI out for any external rig.

### 10.1 Default protocol

- **Output protocol:** **MIDI 1.0 single-channel + 14-bit pitch-bend** at **±2 semitones** range (~0.024 cent resolution per bend step).
- **Pitch-bend timing:** **Strategy B** — bend at `t-1 ms`, note-on at `t`. The 1-ms gap guarantees pitch-bend arrives before note-on across all drivers; latency penalty (~1 ms) is well below the 5–10 ms human perception threshold.
- **RPN preamble:** sent on output-device-connect, configuring the receiver's pitch-bend sensitivity to ±2 semitones (CC 101=0, CC 100=0, CC 6=2, CC 38=0, then null RPN CC 101=127, CC 100=127). Sent again on user-changed sensitivity. Idempotent.

### 10.2 Default key-mapping

**Hybrid (white-key canonical scale + black-key override-palette shortcuts).** White keys carry the maqam's 7-pitch ascending scale starting at the karar MIDI note (default = MIDI 60 / C4). Black keys carry the most-cited regional alternates for adjacent white-key degrees.

**Specific black-key mapping** (for a 7-pitch maqam with karar at C, octave-cyclic):

| Black key | Maps to | Behaviour |
|---|---|---|
| C♯ | First override of degree 1 (typically a subtle nudge — e.g. for Turkish AEU, ±1 koma neighbour) | One-shot retune of the next white-key C. |
| D♯ | First override of degree 2 (e.g. Aleppine 2nd → Cairene 2nd if the maqam ships an alternate; else neutral nudge) | One-shot retune of the next D. |
| F♯ | First override of degree 4 (e.g. Saba's compressed 4th) | One-shot retune of next F. |
| G♯ | First override of degree 5 | One-shot retune of next G. |
| A♯ | First override of degree 6 | One-shot retune of next A. |

Maqamat with fewer override candidates leave the corresponding black keys silent (mapped to `x` in the underlying .kbm). v2 may add a "linear" mode (every key = next maqam degree) and a "karar-anchored" absolute mode.

The user-configurable mapping mode (in MIDI panel): `Hybrid` (default), `Linear`, `Karar-anchored absolute`.

### 10.3 Pitch-bend math

For a 7-pitch maqam with karar at MIDI 60, hybrid mapping, ±2 semitone bend:

```
midi_in = note_number              (the key the user pressed)
degree  = mapping[midi_in]          (1..7 white-key, or "override-of-N" black-key)
target_cents = pitch_table[degree]  (cents-from-karar)
nearest_12edo = round(target_cents / 100) × 100
remaining_cents = target_cents − nearest_12edo
midi_out = 60 + (nearest_12edo / 100)
bend14 = round((remaining_cents / 200) × 8191) + 8192   (clamp 0..16383)
lsb = bend14 & 0x7F;  msb = (bend14 >> 7) & 0x7F
```

Wire (channel 0):
1. `0xE0, lsb, msb` at `t - 1 ms`.
2. `0x90, midi_out, velocity` at `t`.

For note-off, no bend reset (saves 3 bytes; subsequent note-on overwrites).

### 10.4 Drone MIDI channel

- **Default channel: 10** when lead is on channel 1.
- **Configurable** in the MIDI panel; validates `drone-channel ≠ lead-channel`.
- **Pitch-bend follows the karar.** During a Tab modulation, the translator schedules ~30 pitch-bend messages on the drone channel via `MIDIOutput.send(data, timestamp)` at exponentially spaced timestamps over the 600 ms glide (50 Hz update rate). MIDI bandwidth: ~150 bytes/sec on the drone channel — trivial vs. the channel's ~3125 byte/sec ceiling.
- **Drone MIDI output is OFF by default.** The audio drone is on; MIDI drone-out is opt-in.

### 10.5 User-configurable settings

Three settings the user **must** control (no defaults that work for everyone):

1. **Input device** — which MIDI keyboard.
2. **Output device + protocol** — where the maqam-tuned MIDI goes.
3. **Karar MIDI note** — which physical key plays the karar (default 60 / C4; slider range 24–96).

Three settings with sensible defaults:

1. **Mapping mode** → Hybrid (default).
2. **Pitch-bend sensitivity** → ±2 semitones (default).
3. **Live retune held notes** → ON (default; mirrors qanun-mandal-style behaviour). When the karar changes (via Tab modulation in re-center mode), held notes glide to the new mapping over 600 ms.

Other configurable settings (advanced):

- Pass-through input pitch-bend (default ON).
- Reset pitch-bend on note-off (default OFF).
- Drone MIDI output channel (default 10).
- Drone pitch-bend range (default ±2 semitones, matches lead).

### 10.6 File export

- **`.scl + .kbm` bundle** for any active preset. Generated on demand from a "Export tuning" button in the settings drawer. Saves as `<preset_id>.scl` + `<preset_id>.kbm` in a single zip.
- `.scl` format follows Huygens-Fokker conventions: description line, pitch count `N`, `N` ascending pitches in cents (skipping unison; ending with `1200.000000`), `! ` comments preserving the perde names and source citation.
- `.kbm` format: 12-of-12 mapping with `karar_perde.midi_note` as middle, scale degree `N` as the formal-octave index, reference frequency = the karar's actual Hz.
- Round-trip target: export → load into ODDSound MTS-ESP Mini → drive an MTS-ESP-aware soft synth (Surge XT, Pianoteq, u-he Diva) → measured pitch must be within ±1 cent of expected.

### 10.7 What v1 MIDI does NOT include

- **MPE.** v2.
- **MTS Bulk SysEx live.** v1.5 (paired with the standalone proxy mode).
- **MTS-ESP runtime IPC.** Never (browser-incompatible).
- **MIDI 2.0 per-note pitch.** v3 / when Web MIDI 2.0 stabilises.
- **`.tun` write.** v2.
- **SonicWeave source export.** v2 (advanced mode only).

### 10.8 Embedded-only delivery

v1 ships the MIDI translator inside the main app at `?tab=_midi` (BeatForge's secret-tab pattern). The standalone "maqam-tuning pedal" proxy mode (`?mode=proxy`) is **v1.5** — a 2-day shell + URL-routing exercise atop v1's already-complete translator backend.

### 10.9 Latency commitment

- **Translator processing per event:** < 0.5 ms target; < 0.2 ms typical (three O(1) lookups + cents → bend14 arithmetic + 6 bytes out). Measured via the Latency Monitor (translator-only mode) in the MIDI panel.
- **End-to-end (input → external synth audio):** dominated by the OS / browser / driver baseline of 5–15 ms. The 1-ms pitch-bend gap adds another 1 ms. Total v1 commitment: **≤ 30 ms total round-trip** at the 95th percentile across Chrome / Edge / Firefox on a 2024-class desktop.

## 11. Pitch-network with button-overrides — the interaction model

Pivot 6's "pitch-network with button-overrides" is the load-bearing UX of how a player accesses regional pitch flexibility without leaving their default mode. It abstracts the qanun mandal: each scale degree carries a small palette of **candidate pitches** the player can switch among.

### 11.1 The interaction sequence

```
1. User presses key K.
   → Engine plays the canonical pitch for K's scale degree in the active maqam.
   → A subtle highlight pulses the key.

2. User presses K and holds for ≥ 500 ms.
   → A radial palette overlay opens, centred on K.
   → The key's note continues sounding at the canonical pitch.
   → 4–6 alternate-pitch buttons appear at ~80 px from centre,
     each labelled with the alternate's region name + cents-from-canonical.

3. User taps an alternate (within the palette).
   → The held note glides over 50 ms to the alternate pitch.
   → The radial menu closes.
   → The alternate is "armed" for one shot: the next press of K plays the
     alternate instead of the canonical, then reverts.

4. User taps Lock toggle in the radial menu (top-centre button).
   → The alternate becomes sticky for that scale degree.
   → All subsequent presses of K (and of K's octave-equivalents on other keys)
     play the alternate pitch.
   → A "lock" badge appears on K.
   → Lock state persists in IndexedDB with the user's session.

5. User taps elsewhere or hits Esc.
   → Radial menu closes without committing.
```

## 12. Tab-to-current-pitch karar-modulation gesture

Pivot 9's signature improvisational move. As far as research can find, no commercial maqam software does this.

### 12.1 Trigger

- **Keyboard:** `Tab` key, with the playing surface focused. Handler calls `preventDefault()` and checks `document.activeElement` is the playing surface — otherwise a user tabbed into a settings dropdown would trigger karar-modulation instead of moving focus.
- **GUI:** `[↻ Karar]` button in the drone strip. Primary on mobile (no hardware Tab key). Always available.
- **Rebindable** in settings (advanced) — for users with focus-handling conflicts or non-QWERTY layouts.

### 12.2 Behaviour

```
Pre-conditions:
  - Drone is on (else Tab is a no-op).
  - pitchHistory[lastIndex] exists (else Tab plays a soft "ready" pluck).

On Tab press:
  P_drone   = current drone-fundamental pitch (Hz)
  P_target  = pitchHistory[lastIndex].pitch (Hz; the most-recent melody note-on)
  glide_ms  = user-configured glide time (default 600 ms; 200–1500 range)

  1. Cancel any active drone-pitch ramps on the AudioParam.
  2. Schedule:
       droneFundamental.setValueAtTime(P_drone, ctx.currentTime)
       droneFundamental.exponentialRampToValueAtTime(P_target, ctx.currentTime + glide_ms/1000)
  3. All voices in the active interval mode (sa-pa, sa-pa-octave, etc.)
     glide together, interval ratios preserved.
  4. Karar slider's UI knob glides visually to its new position over the same glide_ms,
     with a brief tradition-accent glow.
  5. If modulation mode = Re-center:
       - Set karar = P_target.
       - Refresh per-pitch overrides palette (Aleppine vs Iraqi vs Cairene
         candidates rebase to new karar).
       - Emit `karar_change` event on the bus.
       - MIDI translator updates its mapping; live retune held notes (if enabled).
     If modulation mode = Float:
       - Drone moves; maqam pitch-network unchanged.
       - Override palette unchanged.
       - MIDI mapping unchanged.
```

### 12.3 Repeat-Tab semantics

If Tab fires during an active glide: the active glide is cancelled; a new glide originates from the current (mid-glide) value, targets the new most-recent pitch, runs for `glide_ms`. Monotonic in cents-space; rapid Tab-Tab-Tab feels like a smooth polyglide.

### 12.4 Pitch-history ring buffer

`HISTORY_SIZE = 16` entries, each `{ pitch_hz, timestamp_ms, scale_degree, held }`. Captured on every lead-voice note-on (regardless of source: keyboard, MIDI, QWERTY). Tab uses `pitchHistory[lastIndex]`. With no history (cold session), Tab plays a soft "ready" cue and is otherwise a no-op.

### 12.5 Held-note interaction

- **Re-center + held lead note:** held note keeps playing at its old absolute pitch. If "Live retune held notes" is on (default), the held note glides to the new mapping over the same 600 ms; if off, the held note remains until release.
- **Float + held lead note:** trivial; nothing moves.

### 12.6 Visual feedback

- Karar slider knob glides to new position over `glide_ms` with a tradition-accent glow.
- Override-palette badges (in the per-pitch drawer) animate-pulse during refresh in Re-center mode.
- Subtle, not flashy — the gesture should feel signature without being theatrical.

## 13. Tuning-data layer schema (TypeScript-style prose)

Restated from `research/tuning-systems-formalization.md` §5. Engineers translate to TypeScript types or a Zod schema; what follows is the prose contract.

### 13.1 `MaqamPreset`

The top-level preset object.

- `format_version` — string; required. v1 = `"1.0.0"`. Loader refuses files newer than it understands.
- `id` — string; required. Stable opaque identifier (suggested: `<tradition>.<region>.<maqam>.<variant>`).
- `name` — object: `{ canonical: string, native?: string, transliterations?: string[] }`. `canonical` is the project-official Latin Romanization; `native` is in tradition-correct script; transliterations is a list of accepted alt-spellings for search.
- `tradition` — closed enum: `"turkish" | "arabic_levantine" | "arabic_egyptian" | "persian"`. (v2 adds `"azeri" | "shashmaqam" | "maghrebi" | "hindustani" | "byzantine"`.)
- `region` — string; optional free-form (`"istanbul"`, `"aleppo"`, `"cairo"`, `"baghdad"`, `"tehran"`, …).
- `era` — string; optional. Provenance tag (`"ottoman_classical"`, `"19c_arabic"`, `"radif_qajar"`, `"contemporary"`).
- `karar_perde` — object: `{ name: string, midi_note?: int, cents_offset_from_a440: float }`. The root pitch.
- `pitches` — array of `Pitch`; required. Length 7 typically. Ordered ascending. Stored *with* unison at index 0 (cents 0.0); *without* explicit period (1200 implicit unless `period_cents` on preset, v2).
- `pitches_descending` — array of `Pitch`; optional. Present only when descending diverges (Hüseyni, Saba).
- `ajnas` — array of `JinsRef`; optional. Arabic-only modular composition.
- `seyir` — string; optional free-form prose describing melodic development.
- `aliases` — string[]; optional. Cross-tradition equivalents for search.
- `parent_dastgah_id` — string; optional. Persian-only; flat pointer to a parent dastgāh. (v2: parent becomes a real `Dastgah` object owning `gushehs[]`.)
- `references` — array of `Reference`; optional.
- `tags` — string[]; optional (`"common"`, `"rare"`, `"pedagogical"`).
- `license` — string; optional. Default `"CC0-1.0"` for community presets.
- `author` — string; optional.

### 13.2 `Pitch` (per-degree pitch entry)

- `degree` — integer; 1-indexed scale degree (1 = unison, 2..N = scale degrees).
- `name` — string; required. Traditional perde / pitch name (`"Çârgâh"`, `"Sikah"`, `"E koron"`).
- `name_native` — string; optional native-script form.
- `cents_from_karar` — float; required. Signed, one decimal place. **The authoritative engine value.**
- `ratio` — string; optional. Source-theoretic ratio (`"5/4"`, `"81/64"`) stored as string to avoid float drift. Engine ignores for playback unless the user opts into ratio-exact mode (v2).
- `practice_range_cents` — `[min: float, max: float]`; optional. Empirically observed performance range. Used for UI hints + the override-palette candidate seeds.
- `comma_count` — integer; optional. Turkish-only AEU integer count of Holdrian commas above karar (or above previous perde, depending on context — documented per preset).
- `display_accidental` — enum string; optional. Tradition-keyed:
  - Turkish: `"natural" | "1c_flat" | "4c_flat" | "5c_flat" | "8c_flat" | "1c_sharp" | "4c_sharp" | "5c_sharp" | "8c_sharp"`.
  - Arabic: `"natural" | "half_flat" | "flat" | "half_sharp" | "sharp"`.
  - Persian: `"natural" | "koron" | "sori" | "flat" | "sharp"`.
- `enharmonic_aliases` — string[]; optional cross-tradition spellings.

### 13.3 `Jins` and `JinsRef` (Arabic ajnas-stacking model)

A `JinsRef` (inside a maqam preset's `ajnas[]`):

- `jins_id` — string; required. References a separate library record.
- `transposition_cents` — float; required. Cents above karar at which the jins begins.
- `role` — enum: `"foundation" | "branch" | "alternate"`.

A `Jins` definition (separate library `JinsLibrary.json`, eagerly loaded at app startup):

- `id`, `name`, `name_native`, `aliases` — identification.
- `tradition` — always `"arabic_*"` for v1 (jins-stacking is Arabic-native).
- `intervals_cents` — float[]; the interval *sizes* within the jins (not absolute pitches). Example: Rast jins = `[200.0, 150.0, 150.0]` (sums to 500 = perfect 4th); Hijaz jins = `[150.0, 250.0, 100.0]`.
- `ghammaz_offset_cents` — float; the offset within the jins where the next jins starts when this one is foundation.

The loader, on Arabic preset load, computes `pitches[].cents_from_karar` from `ajnas[]` if `pitches[]` is absent; both can coexist for cross-validation.

### 13.4 `MandalState` (instrument session state, derived from preset + instrument)

A `MandalState` is in-app session state, **not part of a preset file**. Derived from `(MaqamPreset, QanunInstrument) → MandalState`.

- `courses` — `CourseState[]`, length = number of physical courses (typical: 26 Turkish, 27 Arabic).
- For each `CourseState`:
  - `course_index` — int; 1-based.
  - `nominal_pitch_class` — string (e.g. `"D"`, `"Dügâh"`).
  - `available_pitches` — `MandalPosition[]`. The wired-in mandal offsets. Each is `{ index: int, cents_offset_from_nominal: float, name?: string, is_engaged: bool }`.
  - `active_mandal_index` — int. Currently sounding. Mutates as user engages mandals.

Loading a maqam preset onto a qanun walks `pitches[]`, finds for each pitch the course bearing that pitch class, and selects the `available_pitches[]` entry whose offset matches. Mismatches surface a "this preset can't be played on this instrument without retuning" warning with the affected pitches listed.

### 13.5 `KararSpec`

The runtime-active root specification. Not directly serialized (derived from `MaqamPreset.karar_perde` plus user overrides).

- `name` — string (perde name).
- `midi_note` — int (0..127).
- `cents_offset_from_a440` — float (signed; lets the engine compute Hz directly).
- `nudge_cents` — float; the user's per-session ±100-cent nudge from the karar slider. Defaults 0.0.

Effective Hz: `Hz = 440 × 2^((cents_offset_from_a440 + nudge_cents) / 1200)`. For pitches: add `pitch.cents_from_karar` inside the exponent.

### 13.6 `Reference`

```
{
  type: "recording" | "book" | "article" | "url" | "database",
  citation: "Marcus, Scott (2007). Music in Egypt. Oxford UP, ch. 3.",
  url?: "https://maqamworld.com/en/maqam/rast.php",
  notes?: "Marcus's measurement of Cairo Rast Sikah at ~350c."
}
```

### 13.7 Tradition-determined snap grids

- `tradition: "turkish"` → snap to multiples of 22.6415 cents (53-EDO Holdrian) for mandal positions in the per-pitch overrides drawer.
- `tradition: "arabic_*"` → snap to multiples of 50 cents (24-EDO).
- `tradition: "persian"` → no snap; continuous cents.
- The user can always override the snap grid in the advanced editor.

### 13.8 Format versioning

- **Patch (1.0.x):** docs only.
- **Minor (1.x.0):** add new optional fields; loaders skip unknown fields without error.
- **Major (x.0.0):** incompatible; loader refuses and surfaces migration UI.

The loader implements a chain `1.0 → 1.1 → ... → current`. Files older than the chain trigger one-time conversion on import.

## 14. URL share-link encoding

Per Pivot 3 (BeatForge fork) and `research/tuning-systems-formalization.md` §"Sharing model — URL pattern". One core constraint: a 7-pitch preset must compress to under 2 KB.

### 14.1 URL format

```
https://cemergin.github.io/makam_studio/?v=1&p=<base64url(JSON)>
```

- `v=1` — schema version. Forward-compatible; v1 readers refuse v2 payloads with a clear message.
- `p=...` — base64url-encoded JSON of the share payload below. URL-safe alphabet (`-` and `_` instead of `+` and `/`); no padding.

### 14.2 Share-payload JSON shape

```
{
  "tradition": "turkish",
  "maqam_id": "turkish.classical.hicaz",
  "karar": { "name": "C", "midi_note": 60, "nudge_cents": 0.0 },
  "voice_id": "qanun",
  "drone": {
    "voice_id": "vaporwave-choir-tubular-wind",
    "interval": "sa-pa",
    "modulation_mode": "re-center",
    "on": true,
    "volume": 0.30
  },
  "mandal_overrides": [
    { "degree": 3, "alternate_id": "aleppine_sikah" }
  ],
  "label_mode": "perde",
  "layout": "split"
}
```

Optional fields are omitted when at default to minimise payload size. A typical share is < 400 bytes pre-base64 / < 600 bytes after.

### 14.3 Compression and length cap

- For shares > 1.5 KB: gzip-encode then base64url. Adds a `c=gz` parameter so the loader knows to decompress.
- Hard cap: 2.0 KB total URL length. If exceeded, the share dialog warns and offers `Copy as JSON file` instead.

### 14.4 Backward-compatibility rules

- v1 readers ignore unknown fields silently (forward-compat within `v=1`).
- v1 readers refuse `v=2+` URLs and show: "This share was made with a newer makam_studio. Update or open the original share."

## 15. IndexedDB schema (Dexie)

Forked structurally from BeatForge's `BFDatabase`, renamed and extended.

```
class MSDatabase extends Dexie {
  presets!:       Table<MaqamPreset, string>;       // bundled v1 catalog (read-only mirror)
  customMaqams!:  Table<MaqamPreset, string>;       // user-edited / user-created
  sessionState!:  Table<SessionStateRow, string>;   // single-row table keyed by 'current'

  constructor() {
    super('makam_studio');
    this.version(1).stores({
      presets:      'id, tradition, region',
      customMaqams: 'id, tradition, createdAt, updatedAt',
      sessionState: 'key',
    });
  }
}
```

### 15.1 `presets` table

- **Purpose:** read-only mirror of the bundled `MaqamPreset` catalog. Indexed by `id`, `tradition`, `region`.
- **Population:** built once at app startup from `import.meta.glob('./presets/seed/**/*.json', { eager: true })` (BeatForge pattern); the build pipeline ships all 24 v1 presets as per-file JSON in `app/src/presets/seed/<tradition>/<id>.json`.
- **User mutation:** none. Editing forks a copy into `customMaqams`.

### 15.2 `customMaqams` table

- **Purpose:** user-edited copies of seed presets, plus user-created presets (e.g. saved from a long-running override-locked session).
- **Schema:** same as `MaqamPreset` plus `user: true`, `createdAt: number`, `updatedAt: number`.
- **Indexes:** `id`, `tradition`, `createdAt`, `updatedAt`.
- **Constraint:** `id` must not collide with seed `presets.id`; the UI auto-suffixes (`<original_id>.user.<timestamp>`).
- **Backup / restore:** export to JSON file, restore from JSON file. Single-button in settings.

### 15.3 `sessionState` table

Single-row table keyed by `'current'`. Persists the user's last-known UI state for resume-on-refresh.

```
SessionStateRow = {
  key: 'current',
  tradition: TraditionId,
  active_maqam_id: string,
  voice_id: VoiceId,
  karar: KararSpec,
  drone: DroneStateSnapshot,
  layout: LayoutMode,
  label_mode: LabelMode,
  octave_offset: int,
  overrides: { [degree: number]: { alternate_id: string, locked: bool } },
  midi: { mapping_mode: 'hybrid'|'linear'|'absolute', karar_note: int, ... },
  updatedAt: number
}
```

Rewritten on every meaningful state change (debounced ~250 ms).

### 15.4 localStorage usage

Small, frequently-read flags only (BeatForge convention):

| Key | Content |
|---|---|
| `ms_layout` | `'split' \| 'single' \| 'qanun'` |
| `ms_voice` | `'qanun' \| 'baglama' \| 'tar' \| 'ney' \| 'clarinet'` |
| `ms_label_mode` | `'perde' \| 'latin' \| 'cents' \| 'hidden'` |
| `ms_drone_on` | `'true' \| 'false'` |
| `ms_drone_modulation_mode` | `'re-center' \| 'float'` |
| `ms_octave_offset` | int as string |
| `ms_midi_keymap_mode` | `'hybrid' \| 'linear' \| 'absolute'` |
| `ms_midi_karar_note` | int as string |
| `ms_midi_protocol` | `'pitch-bend-1ch' \| 'mts-scl-export'` |
| `ms_midi_active_inputs` | JSON array of device IDs |
| `ms_install_prompt_dismissed_at` | number |
| `ms_safari_acknowledge` | `'true' \| 'false'` |
| `ms_theme` | `'warm-light' \| 'auto'` (only one v1 theme; field present for future) |

## 16. Performance targets

| Target | Commitment |
|---|---|
| **First paint** | < 1.5 s on mid-range phone (iPhone 12 / Pixel 6 class) over 4 G. Budget: ~280 KB gzipped first-load (mirroring BeatForge). |
| **Audio latency** | Total ≤ 30 ms at 95th percentile (input → audio output) on Chrome / Edge / Firefox desktop. Includes Web MIDI input baseline (5–15 ms), translator processing (< 0.5 ms), pitch-bend gap (1 ms), AudioWorklet output buffer (~10 ms). |
| **Pitch precision** | ≤ 1 cent error at A4 across the full v1 preset catalog when round-tripped through .scl/.kbm export → MTS-ESP Mini → Surge XT → measured at audio output. |
| **MIDI translation latency** | < 0.5 ms processing per event (target < 0.2 ms) on top of OS MIDI baseline. Measured via the in-app Latency Monitor. |
| **Polyphony — phone** | ≥ 6 simultaneous voices (Karplus-Strong-equivalent units) on iPhone 12 / Pixel 6, headroom for v2. v1 ships monophonic — this is a v2-readiness target, not a v1 ship-blocker. |
| **Polyphony — desktop** | ≥ 24 simultaneous voices on a 2024-class CPU. v2-readiness. |
| **Bundle size** | < 300 KB first-load gzipped (App JS + CSS + 24 preset JSON + jins library + 5 voice DSP + drone DSP + Dexie + Workbox runtime). Fonts subset-loaded. |
| **PWA install** | Functional offline-first after first load. |

### 16.1 Out-of-budget call-outs

- The drone subsystem alone is ~3 × baseline CPU; with 1 lead voice that's ~4 × on phone — comfortable. v2 polyphony is where CPU becomes the constraint; bowed strings (~5–15 × baseline per voice) explain why v1 defers them.
- Persian gusheh-level pitch detail (per-gusheh pitch shifts) is deferred to v2; v1's daramad-level approximation (§7.3) is stable but loses the radif's micro-variation.

## 17. Accessibility

WCAG 2.1 AA target. Specific commitments:

### 17.1 Keyboard navigation

- **Full app navigable via keyboard.** Tab / Shift-Tab moves focus through: tradition tabs → maqam grid (4×4, arrow-key navigation within) → karar slider → per-pitch overrides drawer → layout toggle → playing surface (right-zone keyboard) → drone controls → settings.
- **Inside the playing surface:** Arrow keys shift visible window or step focus across keys; Space / Enter triggers note-on.
- **Tab is overloaded** for karar-modulation when the playing surface is focused. The handler `preventDefault`s only when the playing surface is `document.activeElement` — otherwise Tab moves focus normally. This is critical for keyboard accessibility and is the single dual-binding.
- **Esc** closes overlays (override-palette, drone advanced, settings).

### 17.2 Screen-reader labels

- Every key in the right zone has an accessible name combining perde + Latin + cents:
  - Example: `aria-label="Sikah, E half-flat, 50 cents flat from E natural, scale degree 3"`.
- Microtonal accidental glyphs have `aria-label` per glyph: `"koron, lowers pitch by ~40 cents"`, `"sori, raises pitch by ~30 cents"`, `"AEU 5-koma flat, lowers pitch by ~113 cents"`.
- The override-palette radial menu announces each alternate by name + region + cents: `"Iraqi Sikah, 360 cents from karar, +10 cents from default"`.
- Drone state changes (Tab modulation in re-center mode) announce the new karar: `"Karar modulated to D, Hicaz on D"`.

### 17.3 RTL handling

- Arabic and Persian labels use `dir="rtl"` containers. The schema's `name_native` is rendered in tradition-correct RTL flow.
- The maqam grid reading order remains left-to-right (the grid is a 2-D array, not flowing prose); but the `name_native` subtitle inside each pad is RTL-formatted.
- The settings drawer detects user OS language preference and flips top-level layout (left/right zones swap) when the active locale is Arabic or Persian. v1 ships UI strings only in English; native-script labels for content (perde, maqam names) are RTL-rendered in their own locales regardless of UI language.

### 17.4 Color contrast

- WCAG 2.1 AA: 4.5:1 for normal text, 3:1 for large text (≥18 pt or 14 pt bold).
- The palette in §18 has been checked against the Wong palette + warm-cream background for compliance.
- All semantic state (active maqam, locked override, depressed key) is signalled with both colour AND shape/icon (per the design-direction "color-blind-safe" requirement).

### 17.5 Reduced motion

Users with `prefers-reduced-motion: reduce`:

- Drone tremolo amplitude reduces from 10 % to 2 %.
- Tab-glide visual effect (the karar slider knob's animated journey) is replaced with a discrete jump.
- Vowel-cycle in the drone choir layer slows to 30 s/cycle (from 10 s) with reduced amplitude.
- All non-essential transitions (pad highlight pulse, pitch-bend ramps in the UI) are replaced with static state changes.

The audio (pitch glide itself) is NOT reduced — that would break the gesture's musical meaning. Only visual motion reduces.

### 17.6 Other commitments

- Focus rings: 2 px coral outline on all focusable elements; 3 px on keyboard-focused right-zone keys.
- Skip-to-content link at the top of the page.
- Form fields (settings drawer) carry explicit `<label>` elements.
- All icons have visible-text or `aria-label` equivalents.
- The Safari / no-Web-MIDI fallback notice is announced via `role="status"` once on first load, then dismissable.

## 19. What is NOT in v1

The seductive paths and why we reject them in v1:

- **No polyphony.** Monophonic v1 simplifies the engine, MIDI translator, and override semantics. Polyphony is v2.
- **No bowed strings (violin / kamancha / lyra).** Real-time Helmholtz-friction modelling at < 1 ms in a Web Audio worklet is a 1–2 month DSP project on its own. The path is Faust → WebAssembly Worklet via Faust-STK's `violinModel`. v2.
- **No tanbur, dutar, zurna.** Tanbur's 30-second sustain breaks K-S stability assumptions; dutar's drone-+-melody character needs polyphony; zurna's loudness-balance is a mixer problem we don't solve in v1. v2 (alongside bowed strings).
- **No MTS-ESP runtime.** Browser-incompatible; MTS-ESP is a desktop-plugin shared-memory IPC. We export `.scl + .kbm` files for MTS-ESP Mini consumption; runtime IPC will never be in browsers. (.scl/.kbm export covers the use case for 95% of MTS-ESP users.)
- **No MPE.** Per-note channels / pitch-bend become useful with polyphony. v2.
- **No MTS Bulk SysEx live output.** v1.5 (paired with the standalone proxy mode).
- **No MIDI 2.0.** No Web MIDI 2.0 spec yet (W3C-stage as of April 2026). v3+ when browser support stabilises.
- **No standalone "maqam-tuning pedal" proxy mode** at `?mode=proxy`. The embedded MIDI translator at `?tab=_midi` is in v1; the polished standalone shell is v1.5.
- **No Hindustani / Byzantine / Maghrebi / Azeri / Shashmaqam preset tabs.** The schema accommodates them now (the `tradition` enum extends in v2), but the preset library and UX strings are v2 / v3.
- **No Egyptian as a separate tab.** v1 merges Egyptian and Levantine into the AR tab via the `region` field on each preset (`"cairo"` vs `"aleppo"` vs `"baghdad"`); v2 may split if user feedback warrants.
- **No practice mode** (perde recognition, makam-identification ear-trainer, speed trainer for melodic motifs). v2.
- **No patch editor.** Voices ship as preset patches with 3–5 user-exposed parameters (§8). Deep parameter editing is a v3 advanced editor.
- **No modern-synth patch category** (vaporwave pluck / synthwave lead / ambient pad / future-funk bass / IDM pluck / dub bass / future R&B keys). The drone *voice* uses vaporwave aesthetics, but the lead-voice modern-synth roster is v1.5+.
- **No social features.** No accounts, no follows, no comments. URL-share is the only social primitive. Permanent.
- **No telemetry, no analytics, no cloud sync.** Permanent.
- **No bundled audio samples; no user sample import.** Permanent (synthesis is constitutive, per MANIFESTO §4).
- **No DAW features.** No mixer console, no multi-track recording, no timeline, no MIDI file import/export. (BeatForge handles rhythm; makam_studio handles pitch; compose externally.) Permanent.
- **No project naming reframe.** "makam_studio" stays through v1; possible reframe to `modal_studio` / `tonal_studio` only if Hindustani / Byzantine ship as first-class in v2/v3.
- **No `.tun` write export.** v2.
- **No SonicWeave source export.** v2 (advanced mode only).
- **No ratio-exact playback mode.** Default cents-from-karar storage and playback. v2 candidate for Yarman 79-tone-purist users.
- **No moving-ison Byzantine drone mode.** PitchHistory infrastructure exists in v1; rest-tone detection is v2 (also culturally fraught — different protopsalti choose different ison points).
- **No 4th tradition tab in v1's UI.** The `[+]` tab is grayed-disabled.
- **No "qanun click" sound effect by default.** The mandal-click sound is shipped, default off, configurable. (User preference; some find it kitsch, some essential.)
- **No PostHog, Sentry, or any third-party SDK.** Permanent.

## 20. Open questions remaining for user input

These are explicitly unresolved by the research and require the user (Cem) to decide before / during build. They are enumerated by impact.

1. **Default Turkish tuning grid: 53-EDO theoretical or 72-EDO performance practice?** v1 ships 53-EDO theoretical (per the schema in §13; preset cents from §7.1 follow AEU theoretical positions). The empirical-vs-theoretical 30-cent gaps (Karaosmanoğlu's f0 histograms) are exposed via `practice_range_cents` and the override palette. **Should v1 also ship a "performance-practice" toggle that shifts each preset to Karaosmanoğlu-measured pitches by default?** Punted to a per-preset metadata flag in v1, default-off. Cem to confirm.

2. **Drone-on-by-default policy across user types.** The research argues for default-on (the larger fusion / Hindustani / Byzantine / Sufi / folk audience reaches for this tool). Classical Ottoman / Arabic / Persian / Maghrebi traditions don't use continuous drone. **Should the user's first-launch experience include a one-shot "Are you classical or fusion?" prompt?** v1 default: no prompt; drone on; one-click off; about page acknowledges the cultural distinction. Cem to confirm this is the right framing.

3. **Mandal click sound default.** Modal click + audio click + both + off; per-tradition acoustic signature. v1 default: subtle off-by-default click (no sound when toggling maqam). The qanun's physical click is part of its character; some users will love it, some find it kitsch. **Default off, with a one-click on toggle in settings? Or default on with a one-click off?** Currently spec'd as default-off; Cem to confirm.

4. **Project naming.** "makam_studio" is accurate for v1 but cramped if Hindustani / Byzantine ship as first-class in v2. The schema already accommodates non-maqam traditions via the `tradition` enum. **Defer reframe ("modal_studio"? "tonal_studio"?) until post-MVP** is the current position. v1 ships as `makam_studio`. Cem to confirm.

5. **Phase 0 closure git tag.** v0.1 (research-complete) vs v0.2 (Phase-1-spec-complete) vs v1.0 (MVP)? Currently spec'd to tag this v1 spec at `v0.2` and reserve `v1.0` for the MVP build. Cem to confirm.

6. **Karar slider granularity.** v1 spec has chromatic 12-EDO + ±100-cent nudge. **Should the chromatic strip use the 17-perde Ottoman tab' choices when Turkish is the active tradition (instead of plain chromatic)?** The research gives no clear answer; v1 ships with plain chromatic + nudge for tradition-uniform UX. Cem to confirm.

7. **What is the reading order of the right-zone keyboard for screen-reader users?** Standard left-to-right is the default; the AT will read each key by its perde + cents label. **For Arabic / Persian active tabs, do we mirror the keyboard so the karar appears on the right and pitches ascend leftward?** Probably no (would break pianist muscle memory); but worth a one-line confirmation. v1 ships keyboard always-LTR; native-script labels remain RTL within each key.

8. **Does v1 share-link encode the full mandal-override state, or just the active maqam + karar?** Currently §14 includes `mandal_overrides[]`. **Worth confirming this is desired vs. just the preset + karar (smaller payload, simpler UX).** v1 ships full mandal-overrides included; Cem to confirm.

9. **Should the v1 catalog drawer (`+ More…` slots in the maqam grid) actually surface the rest of the tradition's repertoire, or should it remain placeholder until v2?** Current spec: drawer exists, lists the full tradition catalog as read-only entries with a "Coming in v2" footer for ungenerated presets. **Alternative: ship only the 8 v1 presets; suppress the drawer entirely.** Cem to confirm.

10. **Default karar pitch on first-launch.** Spec'd at C4 for cross-tradition uniformity (subverts "C is home" minimally). The wave-1 UX review argued for non-C-default (G or D) to subvert "C is home" more aggressively. **Should first-launch karar default to D4 instead?** Currently C4 with a settings option; Cem to confirm.
